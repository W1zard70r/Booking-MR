# Room Booking Service

Сервис бронирования переговорок, разработанный в рамках тестового задания. Система позволяет администраторам управлять ресурсами и расписанием, а пользователям — бронировать доступные временные интервалы (слоты).

## Технологический стек

*   **Язык программирования:** Go 1.25
*   **База данных:** PostgreSQL 15
*   **Библиотеки:**
    *   `chi v5` — маршрутизация HTTP-запросов.
    *   `golang-jwt/jwt/v5` — работа с токенами авторизации.
    *   `lib/pq` — драйвер для PostgreSQL.
    *   `bcrypt` — хеширование паролей.
*   **Инфраструктура:** Docker, Docker Compose.

## Архитектура и ключевые решения

Проект реализован с соблюдением принципов чистой архитектуры (Clean Architecture), что обеспечивает разделение ответственности между слоями бизнес-логики, API и работы с данными.

### 1. Механизм генерации слотов
В системе используется подход прегенерации слотов на 30 дней вперед.
*   **Обоснование:** Согласно техническому заданию, слоты должны иметь стабильные UUID, сохраненные в базе данных. Прегенерация позволяет избежать вычислительной нагрузки на эндпоинты чтения и гарантирует постоянство идентификаторов.
*   **Скользящее окно (Rolling Window):** Реализован фоновый процесс (Background Worker), который запускается при старте сервера и далее ежечасно. Он автоматически догенерирует слоты на будущие даты, поддерживая окно доступности в 30 дней.

### 2. Предотвращение двойного бронирования
Защита от состояния гонки (Race Condition) реализована на уровне СУБД PostgreSQL.
*   **Метод:** Создан частичный уникальный индекс `idx_unique_active_booking` по полю `slot_id`, который учитывает только записи со статусом `active`. Это гарантирует невозможность создания двух активных броней на один и тот же слот даже при одновременных запросах.

### 3. Идемпотентность операций
Операция отмены бронирования является идемпотентной. Повторные запросы на отмену уже отмененной брони возвращают успешный статус (200 OK) с актуальным состоянием объекта, не вызывая ошибок и лишних изменений в базе данных.

### 4. Безопасность и авторизация
*   **JWT:** Авторизация реализована через JSON Web Tokens. Идентификатор пользователя и его роль извлекаются из токена в middleware.
*   **Хеширование:** Пароли пользователей хранятся в виде хешей bcrypt, что исключает хранение учетных данных в открытом виде.

## Инструкция по запуску

1.  **Развертывание контейнеров:**
    ```bash
    docker-compose up --build -d
    ```

2.  **Применение миграций:**
    ```bash
    make migrate
    ```

3.  **Наполнение базы тестовыми данными:**
    ```bash
    make seed
    ```

Сервис будет доступен по адресу: `http://localhost:8080`.

## Тестирование и покрытие кода

Для проверки корректности работы системы реализован набор интеграционных и E2E-тестов, покрывающих основные бизнес-сценарии: от регистрации до сложных случаев отмены брони и работы фоновых процессов.

*   **Запуск тестов с расчетом покрытия:**
    ```bash
    make cover
    ```
*   **Результаты:** Текущее покрытие кода в директории `internal/` составляет более 74%.

## Основные эндпоинты

### Публичные
*   `GET /_info` — проверка доступности сервиса.
*   `POST /dummyLogin` — получение тестового токена (обязательно по ТЗ).
*   `POST /register` — регистрация нового пользователя.
*   `POST /login` — авторизация по email и паролю.

### Для пользователей (роль user)
*   `GET /rooms/list` — список переговорок.
*   `GET /rooms/{roomId}/slots/list` — список свободных слотов на дату.
*   `POST /bookings/create` — создание брони.
*   `GET /bookings/my` — список будущих броней пользователя.
*   `POST /bookings/{bookingId}/cancel` — отмена собственной брони.

### Для администраторов (роль admin)
*   `POST /rooms/create` — создание новой переговорки.
*   `POST /rooms/{roomId}/schedule/create` — настройка расписания (единоразово).
*   `GET /bookings/list` — просмотр всех броней системы с пагинацией.
